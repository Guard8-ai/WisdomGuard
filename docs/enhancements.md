# Enhancement Sections

WisdomGuard adds four sections to a base guide. Each section is generated by a single Gemini API call with a structured JSON schema response.

---

## Common Workflows

**Markdown heading:** `## Common Workflows`

Multi-step bash recipe blocks for realistic tasks. Between 3 and 7 workflows are generated, each with a name and a complete bash script.

**Example output:**

```markdown
## Common Workflows

### Deploy a Rust service to production
```bash
# 1. Run tests first
cargo test --release

# 2. Build the optimized binary
cargo build --release --target x86_64-unknown-linux-musl

# 3. Package and upload
docker build -t my-service:$(git rev-parse --short HEAD) .
docker push registry.example.com/my-service:latest
```

### Add and audit a new dependency
```bash
# Add the dependency
cargo add serde --features derive

# Check for known vulnerabilities
cargo audit

# Verify it builds
cargo check
```
```

**Placement in merge mode:** Inserted before the first `## ` heading that is not `## Quick Reference`.

## Common Mistakes

**Markdown heading:** `## Common Mistakes`

A three-column table of Wrong / Right / Why patterns ‚Äî the most frequent errors made when using the tool.

**Example output:**

```markdown
## Common Mistakes

| Wrong | Right | Why |
|-------|-------|-----|
| `cargo build` in prod | `cargo build --release` | Debug builds are 10‚Äì100x slower and include panic info |
| `cargo update` without testing | `cargo update && cargo test` | Dependency updates can break compilation or behaviour |
| Hardcoding `./target/debug/app` paths | Use `cargo run` or `$(cargo build --message-format=json \| ...)` | Target paths change with `--target` flags |
```

**Placement in merge mode:** Inserted before the `---` footer line.

## Key Commands

**Markdown heading:** `## Key Commands` (standalone mode only)

A bullet list of the top 20% of commands that cover 80% of real use cases. In merge mode this section is omitted ‚Äî the base guide's Quick Reference already serves this purpose.

**Example output:**

```markdown
## Key Commands

- `cargo build --release` ‚Äî production build
- `cargo test` ‚Äî run all tests
- `cargo add <dep>` ‚Äî add a dependency
- `cargo clippy` ‚Äî run the linter
- `cargo fmt` ‚Äî format source code
- `cargo audit` ‚Äî check for security vulnerabilities
```

## Error Messages

**Markdown heading:** `## Error Messages`

A two-column table mapping common error strings to actionable solutions.

**Example output:**

```markdown
## Error Messages

| Error | Solution |
|-------|----------|
| `error[E0308]: mismatched types` | Check the expected type vs the actual type. Use `.into()` or explicit casting if types are compatible. |
| `error[E0502]: cannot borrow as mutable because it is also borrowed as immutable` | Restructure to end the immutable borrow before creating the mutable one. |
| `Blocking waiting for file lock on package cache` | Another `cargo` process is running. Wait for it or remove `~/.cargo/.package-cache.lock`. |
| `RUSTSEC-XXXX-XXXX: ... vulnerability` | Run `cargo update -p <crate>` to upgrade to the patched version. |
```

**Placement in merge mode:** Inserted before the `---` footer line, after Common Mistakes.

## Merge Insertion Points

When `--base-guide` is provided, enhancements are inserted at specific positions:

```
[Quick Reference]           ‚Üê kept intact
                            ‚Üê Common Workflows inserted here
[Command/Endpoint Reference]‚Üê kept intact
[Global Options]            ‚Üê kept intact
                            ‚Üê Common Mistakes inserted here
                            ‚Üê Error Messages inserted here
---                         ‚Üê footer separator
**Format**: ...             ‚Üê kept intact
```

    }

}
```

## LLM Prompt Structure

WisdomGuard sends a structured prompt requesting a JSON response conforming to the `EnhancementResponse` schema. The prompt includes:

- IR type (`cli`, `api`, or `doc`) auto-detected from the JSON fields (`framework`+`commands` ‚Üí cli, `endpoints`+`spec_format` ‚Üí api, `modules`+`source_format` ‚Üí doc)
- Tool/API name and version from the IR
- Number of commands or endpoints (to calibrate output size)
- Instructions for each section with count targets
- The full JSON schema of the expected response

Use `--dry-run` to print the complete prompt without making an API call.
